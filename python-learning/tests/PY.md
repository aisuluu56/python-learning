1. Жизненный цикл и модели разработки ПО
1. Понятие жизненного цикла программного обеспечения (ЖЦ ПО). Основные этапы.
Жизненный цикл ПО — это последовательность всех этапов, которые проходит программный продукт от момента возникновения идеи до его списания и удаления из эксплуатации. Основные этапы:

Анализ требований: Понимание того, что именно нужно разработать.

Проектирование: Создание архитектуры, моделей данных и интерфейсов.

Разработка (кодописание): Непосредственное создание программного кода.

Тестирование: Поиск ошибок и проверка соответствия требованиям.

Внедрение: Установка программы в рабочую среду заказчика.

Сопровождение: Исправление багов, обновление и доработка в процессе использования.

2. Каскадная модель (Waterfall).
Это классическая последовательная модель. Каждый этап должен быть полностью завершен, прежде чем начнется следующий.

Преимущества: Простота управления, понятные сроки, хорошая документация.

Недостатки: Высокий риск (ошибки выявляются в конце), невозможность возврата к предыдущим этапам, позднее получение рабочего продукта.

Область применения: Проекты со стабильными, заранее четко определенными требованиями (госсектор, медицина, критические системы) .

3. Итерационная и инкрементная модели.

Итерационная модель: Разработка ведется циклами. Каждый цикл (итерация) проходит через все этапы ЖЦ (анализ -> проект -> код -> тест), но на каждой итерации качество и функциональность продукта улучшаются.

Инкрементная модель: Продукт создается порциями (инкрементами). Система разбивается на части, которые реализуются и внедряются по одной.

Отличие: Итерация подразумевает «улучшение» существующей модели, а инкремент — «добавление» новой функциональности. Часто эти модели объединяются.

4. Спиральная модель.
Модель, ориентированная на управление рисками. ЖЦ разбивается на витки спирали. Каждый виток включает: планирование, анализ рисков, разработку и оценку заказчиком. Наиболее эффективна для крупных, сложных и новых проектов с высокими рисками.

5. Гибкие методологии (Agile).
Agile — это философия разработки, ставящая во главу угла взаимодействие людей и готовность к изменениям, а не следование жестким планам.

Основные принципы (Agile Manifesto): Люди и взаимодействие важнее процессов; работающий продукт важнее документации; сотрудничество с заказчиком важнее контракта; готовность к изменениям важнее следования плану.

2. Парадигмы и принципы ООП
6. Структурное программирование.
Подход, исключающий использование оператора безусловного перехода goto. Код делится на три базовые конструкции:

Последовательность: Выполнение команд друг за другом.

Выбор (ветвление): Выполнение одного из путей в зависимости от условия.

Цикл: Повторение блока кода.
Декомпозиция осуществляется через функции и процедуры.

7. Объектно-ориентированное программирование (ООП).
Парадигма, основанная на объектах (экземплярах классов), которые объединяют в себе данные и методы их обработки.

Класс: Шаблон или чертеж для создания объектов.

Объект: Конкретный экземпляр класса, созданный в памяти.

Поле (атрибут): Переменная, хранящая данные объекта.

Свойство: Механизм защищенного доступа к полю (позволяет валидировать данные при присваивании).

Метод: Функция, описывающая поведение объекта.

8. Основные принципы ООП.

Инкапсуляция: Объединение данных и кода в одном классе сокрытие внутреннего устройства. Внешний код не может произвольно менять данные объекта, а делает это только через методы.

Наследование: Возможность создавать новые классы на основе существующих, переиспользуя их код и свойства.

Полиморфизм: Возможность использовать объекты разных классов через общий интерфейс (один метод работает по-разному в зависимости от типа объекта).

Абстракция: Выделение главных характеристик объекта и игнорирование второстепенных деталей реализации.

9. Сравнение структурного и ООП подходов.

Структурный фокусируется на функциях и данных раздельно (Top-down), что хорошо для небольших скриптов, но сложно масштабируется.

ООП фокусируется на объектах (Bottom-up), связывая данные с логикой их обработки. Это обеспечивает лучшую расширяемость и поддержку больших проектов за счет инкапсуляции и полиморфизма .

3. Паттерны проектирования
10. Понятие, назначение и классификация паттернов.
Паттерн проектирования — это проверенное временем, типовое решение частой проблемы проектирования . Они ускоряют разработку, делают код более читаемым и поддерживаемым. Классификация (GoF):

Порождающие: Как создавать объекты.

Структурные: Как составлять классы и объекты в более крупные структуры.

Поведенческие: Как объекты взаимодействуют и распределяют обязанности.

11. Порождающие паттерны.

Фабричный метод (Factory Method): Определяет интерфейс создания объекта, но перепоручает решение, какой именно класс создать, подклассам.

Абстрактная фабрика (Abstract Factory): Создает семейства взаимосвязанных или зависимых объектов без привязки к их конкретным классам.

Одиночка (Singleton): Гарантирует, что у класса есть только один экземпляр, и предоставляет к нему глобальную точку доступа .

12. Структурные паттерны.

Адаптер (Adapter): Позволяет объектам с несовместимыми интерфейсами работать вместе. Один объект оборачивается в адаптер, чтобы его вызовы принимались другим объектом .

Декоратор (Decorator): Динамически добавляет объекту новые обязанности через обертку, не затрагивая другие объекты этого класса .

Фасад (Facade): Упрощает интерфейс сложной подсистемы, предоставляя одну простую точку входа для множества операций.

13. Поведенческие паттерны.

Наблюдатель (Observer): Создает механизм «один-ко-многим». При изменении состояния одного объекта (наблюдаемого) все зависимые объекты (наблюдатели) получают уведомление .

Стратегия (Strategy): Позволяет выбирать алгоритм выполнения задачи во время выполнения программы.

Команда (Command): Инкапсулирует запрос в виде объекта, что позволяет параметризовать клиентов, ставить запросы в очередь или поддерживать операции отмены (undo).

4. Архитектура, модульность и асинхронность
14. Событийно-управляемое программирование.
Поток выполнения определяется внешними событиями.

Источник события: Объект, генерирующий событие (кнопка, таймер, ОС).

Событие: Сигнал о том, что что-то произошло (нажатие).

Обработчик события: Метод или функция, которая запускается в ответ на событие для выполнения логики.

15. Принципы разработки событийно-управляемого интерфейса.
Ключевой принцип — отзывчивость. Интерфейс не должен блокироваться на время выполнения длительных задач. Обычно используется цикл обработки событий (Event Loop), который развязывает логику отображения и бизнес-логику.

16. Синхронное и асинхронное выполнение.

Синхронное: Программа выполняет команду и ожидает её завершения, прежде чем перейти к следующей (блокирует поток).

Асинхронное: Программа запускает длительную задачу (например, запрос в сеть) и продолжает работать. Когда задача завершится, результат обработается через колбэк или await.

Назначение асинхронных методов: Повышение отзывчивости UI и пропускной способности сервера за счет неблокирующего ввода-вывода.

17. Проблемы асинхронного программирования.

Состояние гонки (Race condition): Результат зависит от того, в какой последовательности выполняются потоки.

Взаимная блокировка (Deadlock): Два или более потока ждут освобождения ресурсов, которые удерживает друг друг, и никто из них не может продолжиться.

Блокировка потока: Неправильное использование синхронных вызовов в асинхронном контексте.

Сложность обработки ошибок: Исключения в асинхронном коде сложнее отследить и обработать.

18. Контентные приложения.
Приложения, главная цель которых — потребление контента (новости, видео, книги).

Структура: Поставщик данных (Content Provider), кэш, модуль навигации, адаптивный UI.

Особенности: Важна быстрая загрузка, плавность скролла и эффективное использование памяти, так как объем данных велик.

19. Спецификация программного модуля.
Документ, детально описывающий требования к модулю.

Назначение: Фиксация требований для разработчика и тестировщика.

Структура: Функциональные требования, описание интерфейсов модуля, используемые библиотеки, ограничения, сценарии использования.

20. Этапы разработки ПО-модуля.

Изучение Технического Задания (ТЗ).

Написание спецификации модуля.

Проектирование логики и архитектуры модуля.

Реализация (написание кода).

Модульное тестирование.

Интеграция с остальной системой.

21. Модульное программирование.
Программа разбивается на независимые блоки (модули).

Интерфейс модуля: Публичная часть (какие функции доступны).

Реализация: Внутренний код, скрытый от внешних пользователей.
Разделение интерфейса и реализации позволяет менять внутренности модуля, не ломая программы, которая его использует.

22. Связность и связанность модулей.

Связность (Cohesion): Насколько сильно элементы внутри одного модуля относятся к одной цели. Нужно стремиться к высокой связности.

Связанность (Coupling): Насколько модули зависят друг от друга. Нужно стремиться к низкой (слабой) связанности.

Снижение зависимости: Использование интерфейсов, передача зависимостей (Dependency Injection), событийная модель.

5. Обеспечение качества
23. Понятие отладки (Debugging).
Процесс поиска, анализа и устранения ошибок в коде.

Этапы: Воспроизведение ошибки, локализация (поиск причины), устранение, повторная проверка.

Методы: Логирование, статический анализ кода, реверсивная отладка.

24. Инструментальные средства отладки.

Точка останова (Breakpoint): Остановка выполнения программы на определенной строке.

Пошаговое выполнение: Движение по коду построчно (Step Into — войти в функцию, Step Over — пропустить функцию).

Просмотр переменных: Оценка текущего состояния памяти.

Стек вызовов (Call Stack): История вызовов функций, показывающая путь выполнения до текущей строки.

25. Понятие тестирования.
Процесс проверки программного обеспечения для обнаружения ошибок и подтверждения соответствия требованиям.

Цели: Найти дефекты, убедиться в работоспособности.

Принцип: Тестирование показывает наличие ошибок, но не может гарантировать их полное отсутствие.

26. Основные уровни тестирования.

Модульное (Unit): Тестирование отдельных функций или классов.

Интеграционное: Проверка взаимодействия между связанными модулями.

Системное: Тестирование всей системы в целом.

Приемочное (UAT): Проверка заказчиком, готов ли продукт к релизу.

27. Методы тестирования.

«Черный ящик»: Тестирование без знания внутреннего кода. Тестировщик проверяет входные данные и ожидаемые выходные результаты. Применяется для системного и приемочного тестирования.

«Белый ящик»: Тестирование с полным знанием архитектуры и кода. Покрытие кода тестами, проверка ветвлений. Применяется на уровне модулей.

28. Сценарий тестирования и тест-кейс.

Тест-кейс (Test Case): Конкретный набор условий, входных данных, действий и ожидаемых результатов для проверки одной функции.

Структура: Идентификатор, название, шаги, входные данные, ожидаемый результат, фактический результат.

29. Модульное тестирование (Unit-тесты).
Тестирование минимальных проверяемых частей кода (например, одной функции).

Требования к качественному тесту: Быстрота выполнения, отсутствие внешних зависимостей (нет сетевых запросов), изолированность (не зависит от других тестов), воспроизводимость.

30. Рефакторинг кода.
Внутренняя переработка структуры кода без изменения его внешнего поведения.

Цели: Повышение читаемости, упрощение поддержки, снижение сложности.

Приемы: Переименование переменных, выделение повторяющегося кода в отдельные функции, удаление дублирования.

31. Оптимизация ПО.
Повышение эффективности программы.

Производительность: Выбор более быстрых алгоритмов (например, хэш-таблицы вместо списков), профилирование «узких мест».

Память: Использование пулов объектов, очистка неиспользуемых ресурсов.

Читаемость: Чистый код проще оптимизировать в долгосрочной перспективе.

32. Разработка мобильных приложений.

Особенности: Ограниченные ресурсы (батарейка, память), разная производительность устройств, особенности взаимодействия (тачскрин).

Архитектура: Разделение логики (Model-View-ViewModel), кэширование данных для работы офлайн.

Платформы: Нативная разработка (Swift для iOS, Kotlin/Java для Android) или кроссплатформенная (Flutter, React Native).
